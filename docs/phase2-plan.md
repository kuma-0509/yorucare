# ヨルケア Phase 2 計画メモ（アカウント / クラウド保存 / リマインド / 支援者共有）

このメモは、レビューで挙がった事業・技術課題のうち **「Phase 1 のスタック（Next.js + localStorage、バックエンドなし）から大きく踏み出す施策」** をまとめたものです。これらはサーバ・認証・DB の新規導入を伴い、プロジェクト方針「使う技術スタックを勝手に変更しない」に該当するため、**実装の前に意思決定が必要**です。本メモはそのための判断材料です。

> 本メモの段階では、コードとしての着手は **匿名イベント収集エンドポイント `POST /api/events`（同一オリジン）** のみ。継続率の集計に必要な「永続ストア」は未接続で、ここが最初の判断ポイントです。

---

## 0. 最初に決めるべき1つの問い

> **このプロダクトのゴールは「本人が記録できること」か、「定着率が上がること」か。**

- 前者なら、Phase 1 はほぼ完成形。Phase 2 は「継続の仕掛け」中心で足りる。
- 後者（＝事業として誰かが対価を払う）なら、**ソロ日記から「本人同意の共有ツール」へ重心を移す**必要があり、4〜7 すべてが連動する。

この判断が Phase 2 設計全体を規定します。

---

## 1. 計測の永続化（最優先・土台は着手済み）

**現状**: `src/lib/analytics.ts` が匿名 `installId` 付きで `POST /api/events` に送信。`src/app/api/events/route.ts` は受信してサーバログに出すだけ（永続化なし）。

**必要な一手**: イベントを永続ストアへ保存し、継続率（D1 / D7 / D14、週あたり記録日数、初回→2回目保存の転換率）を集計できるようにする。

**候補スタック（Vercel 前提）**:

| 候補 | 長所 | 短所 |
| --- | --- | --- |
| Vercel KV (Redis) | 導入が最小、サーバレスと相性◎ | 集計クエリは自前 |
| Vercel Postgres / Neon | SQL で継続率集計が容易 | スキーマ運用が必要 |
| 外部 Analytics (PostHog 等) | ダッシュボード込み | 健康隣接データの外部送信に同意・規約整理が必要 |

**推奨**: まず **Vercel Postgres / Neon** に `events(install_id, name, at, meta, received_at)` の1テーブル。`route.ts` の `TODO(phase2)` 箇所に INSERT を足すだけで動く構造にしてある。

**合否ライン例**: パイロット参加者の Day7 継続 40% 以上。

---

## 2. アカウント + クラウド保存

**目的**: 端末紛失・機種変・ブラウザのデータ削除（iOS Safari の ITP 等）でも記録が消えない状態にし、初めて「数週間続くか」を正しく検証できるようにする。

**設計の勘所**:

- データは **要配慮個人情報**（健康情報）。取得・保存には明示同意、保存時の暗号化、削除権（アカウント削除＝データ削除）をセットで設計する。
- `src/lib/repository.ts` を **インターフェース化**し、`localStorage 実装` と `API 実装` を差し替え可能にする。
- 移行時の最大の落とし穴は **同期/非同期境界**。現行 repository は同期 `Result<T>`。API 化で全メソッドが `Promise<Result<T>>` になり、呼び出し側（`today-record-tab` など）が連鎖的に破壊的変更を受ける。**Phase 1 のうちにシグネチャを `Promise` 前提へ寄せておく**と移行が機械的になる。

**候補**: Supabase（Auth + Postgres + RLS）。行レベルセキュリティで「本人のデータは本人のみ」を担保しやすい。メール/マジックリンク中心でパスワード負荷を下げる。

> 注: Supabase 等の導入は依存とスタックの追加。採用時は理由（認証 + RLS + DB を一体で最小工数）を明記してから着手する。

---

## 3. リマインド（継続レバー）

**目的**: 「夜、思い出す」仕掛け。毎日習慣プロダクトの最大の継続レバー。

**候補**:

- **Web Push**: 追加コストなしだが iOS Safari は PWA インストール必須で到達率に難あり。
- **LINE 連携（推奨の現実解）**: 日本の対象層に最も届く。LINE Messaging API で夜の定時通知。`smartphone-test-checklist.md` でも Phase 2 候補として既出。

**最小実装**: 通知 ON/OFF と時刻設定のみ。文面は「責めない」トーン（COPY のボイスに合わせる）。

---

## 4. 本人同意ベースの支援者共有（＝事業価値の発生点）

**目的**: 週次サマリーや「しんどさのサイン」の傾向を、**本人が選んで** 支援員・産業医・上長に共有。定着支援の現場価値は支援者の介在で生まれるため、ここが収益化の起点。

**鉄則**:

- **デフォルト非共有・本人主導（オプトイン）**を厳守。評価への悪用懸念があるため、共有範囲・期間・項目を本人が都度選べること。
- 共有は「閲覧専用リンク」や「期間限定スナップショット」から始め、常時連携は後段。

---

## 5. 危機時セーフティネット（倫理の最低ライン・優先実装可）

メンタル隣接プロダクトとして、「しんどさのサイン＝あり」記録時などに **相談窓口リンク**（いのちの電話・自治体窓口等）を常設表示する。これは 2〜4 を待たずに Phase 1 系の小改修としても入れられる。

---

## 6. 事業モデル仮説（要検証）

- 本人課金は成立しにくい → 無料が基本線。
- 本命は **B2B2C**:
  1. 就労移行支援 / 就労継続支援事業所（定着支援が報酬・成果に直結）
  2. 障がい者雇用をする企業の健康経営・定着部門
  3. 自治体・地域の就労支援（補助金・実証事業＝最初の資金源）
- マネタイズ順序（仮説）: 自治体/支援機関と実証 PoC → 事業所向け SaaS → 企業の健康経営へ展開。

---

## 推奨ロードマップ（Phase 2 内の順序）

1. **計測の永続化**（1）— 継続率が見えないと投資判断ができない。土台は着手済み。
2. **危機時リンク**（5）— 小改修で倫理リスクを下げる。
3. **アカウント + クラウド保存**（2）— 「続くか」を正しく測る前提。
4. **リマインド**（3）— 継続レバー。
5. **本人同意の支援者共有**（4）— 事業価値の発生点。

> 1 → 5 へ進むほど、スタック変更とコンプライアンス負荷が大きくなる。各ステップを「検証 → 次の投資」のゲートにする。
